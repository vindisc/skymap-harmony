# HarmonyOS UI 收口规则

## 1. 当前结论

HarmonyOS `v0.1.0` 已通过华为审核，UI 进入生产冻结状态。生产基线以华为审核通过的安装包、Git 标签 `v0.1.0` 和提交 `bd4fcda` 为准。

从此以后，UI 调整不再以“看起来还能更好”为启动理由。只有达到本文准入条件的变更，才能重新打开已经冻结的页面；通过验收后，必须在同一批次更新证据并再次冻结。

## 2. 基线权威顺序

发生冲突时，按以下顺序判断：

1. 华为审核通过的 `v0.1.0` 安装包：线上用户实际看到的生产基线。
2. Git 标签 `v0.1.0`：可追溯的代码基线。
3. `docs/harmony/main-tabs-ui-baseline-20260628.md` 及其截图：四个主 Tab 的结构和视觉参考。
4. `DesignTokens.ets`、`AppDesign.ets` 和已复用组件：后续实现约束。

根目录临时分析稿、聊天记录、单张局部截图和未进入正式索引的 Markdown 不是实现依据。它们可以提供问题线索，但必须先整理成 UI 变更单。

标签之后尚未提交、尚未验收的工作区内容属于下一版本候选，不自动进入生产基线。

## 3. 允许打开 UI 的四类变更

| 等级 | 准入条件 | 处理方式 |
| --- | --- | --- |
| P0 阻断缺陷 | 页面不可用、按钮不可达、崩溃、关键内容被系统区域或键盘遮挡 | 当前版本立即修复 |
| P1 可用性缺陷 | 可稳定复现的溢出、截断、错位、滚动失败、状态反馈缺失、可访问性问题 | 进入最近一次维护批次 |
| P2 产品变化 | 新功能或信息架构变化导致现有 UI 无法承载 | 先完成产品范围和变更单，再进入下一功能版本 |
| P3 主观微调 | 仅因为更顺眼、更紧凑、换一种颜色、再挪几个像素 | 冻结期不处理 |

P0 / P1 至少要有复现步骤、当前截图和可判断的验收标准。P2 必须先确定功能边界，禁止以 UI 微调代替产品决策。

## 4. 冻结范围

以下内容默认冻结：

- 四个主 Tab 的顺序、信息架构和首屏优先级。
- 页面标题位置、内容起始位置、底部安全区和浮动操作区。
- 字号、行高、颜色、间距、圆角、阴影和动效 Token。
- 卡片层级、列表密度、设置项结构和解释文案密度。
- 编辑页键盘行为、长文本输入和底部操作按钮。
- 横图、竖图、方图的预览与导出结构。
- 2*2、2*4 服务卡片的已锁定样式。

业务功能修改不得顺手调整这些内容。确需修改时，只改变更单中列出的页面、状态和设计维度；同一批次不得夹带无关视觉优化。

## 5. 单一变更流程

1. 使用 `UI_CHANGE_RECORD_TEMPLATE.md` 建立一张变更单，先写问题和验收标准，不先写解决方案。
2. 同一页面的同类问题合并成一个批次，一次完成结构、间距、文案和状态检查。
3. 开发前明确“不改什么”，防止改标题时顺手改卡片，改卡片时顺手改颜色。
4. 实现时优先复用 `DesignTokens.ets`、`AppDesign.ets` 和现有组件；禁止为单页引入近似 Token。
5. 运行统一门禁：`node scripts/verify_ui_closure.mjs`。
6. 按变更单完成真机状态矩阵，保存修改前后截图；自动检查不能代替真机验收。
7. 验收结论、截图、代码和受影响基线在同一批次提交。验收通过即重新冻结，后续再改必须新建变更单。

每张变更单只允许一个明确结论：`接受`、`退回修改` 或 `取消`。不能以“先合并再看看”结束。

## 6. 最低真机验收矩阵

| 页面类型 | 必验状态 |
| --- | --- |
| 四个主 Tab | 默认数据、空状态、长文案、滚动到底、系统手势区 |
| 编辑和表单页 | 键盘弹出、长文本、输入框聚焦、底部按钮、保存失败反馈 |
| 图片和复盘卡 | 横图、竖图、方图、无图、预览和导出一致性 |
| 设置子页面 | 页面顶部、滚动顶部和底部、返回、输入失败状态 |
| 服务卡片 | 2*2 和 2*4 真机桌面、浅色背景、内容刷新、点击跳转 |

只改一个局部，也要验收该局部所在页面的完整首屏和相邻状态，避免局部正确、整体失衡。

## 7. 完成定义

一项 UI 变更只有同时满足以下条件才算完成：

- 有符合准入等级的变更单，不属于 P3 主观微调。
- 范围内所有验收标准可直接判断通过或失败。
- 没有新增近似颜色、字号、间距、圆角或阴影常量。
- 空、加载、成功、失败、长内容和安全区状态没有回归。
- 统一门禁通过，ArkTS 至少编译到 `CompileArkTS`；签名环境失败需单独记录。
- 真机截图与变更单中的目标一致。
- 基线和变更记录已同步，提交不包含范围外 UI 改动。

- `scripts/verify_floating_dock_safe_area.mjs`
