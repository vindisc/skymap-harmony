# Skymap UI Interaction Guide

## 0. 文档目的

本文件定义 Skymap 后续界面迭代的基础设计方向。任何新增页面、按钮、弹层、卡片、统计区、设置表单或导出流程，都应先对照本文件，再进入客户端实现。

Skymap 的界面目标不是“功能都摆出来”，而是让摄影复盘这件事变得安静、清楚、可信。用户打开应用时，应立刻知道下一步该做什么；阅读复盘时，应让照片和判断内容成为主角；导出和同步时，应让结果、位置、失败恢复路径都明确。

## 1. 借鉴对象

### 1.1 平台底线：HarmonyOS / Apple HIG / Material Design

吸收点：

- 使用系统级导航和手势，不在手机端重复放无意义的显式返回按钮。
- 触控目标不低于 44vp，主要按钮和输入框优先使用 48vp。
- 页面底部、键盘弹出、系统手势区必须留安全空间。
- 弹层用于承载一组次级动作，不把低频动作平铺在主页面。
- 按钮、输入框、列表、卡片必须有明确按压和禁用态。

落地要求：

- 顶级页面使用底部导航；二级流程依赖系统返回手势。
- 关键动作必须有可见按钮，不能只靠隐藏手势。
- 危险动作必须二次确认，且和普通动作视觉上分开。

### 1.2 Google Photos：图库、搜索、编辑、分享的主线收敛

吸收点：

- 主线动作很少：进入内容、编辑、整理、搜索、分享。
- 照片和内容占视觉主位，工具按钮不抢戏。
- 高频动作在当前屏可见，低频动作进入菜单或底部弹层。
- 列表和图库用清晰缩略图、标题、时间和状态帮助用户回忆。

落地要求：

- 首页只保留一个最强主动作：开始复盘。
- 复盘库优先强化搜索、筛选、最近记录和记录状态。
- 预览页顶层只放编辑和导出；导出细项进入底部弹层。

### 1.3 Notion：信息架构、分区、空状态

吸收点：

- 页面不是堆控件，而是按任务分区。
- 标题、说明、状态和操作之间层级清楚。
- 空状态不只是“暂无数据”，要给下一步入口。
- 设置页和详情页保持克制，减少装饰，强调可读性。

落地要求：

- 每个页面只允许一个核心目标。
- 页面首屏必须回答：这里是什么、当前状态是什么、下一步能做什么。
- 空状态必须包含一句解释和一个合理行动。

### 1.4 Bear：阅读体验、字体、低干扰按钮

吸收点：

- 内容页应安静、轻、可读。
- 字体层级靠大小、字重、行高和留白组织，不靠大量色块。
- 标签和状态使用轻量底色，不使用高饱和装饰。
- 长文本优先可读，不为了压缩界面牺牲行高。

落地要求：

- 复盘阅读页减少按钮密度，避免把导出、复制、上传等动作全部顶层展示。
- 正文不低于 15fp，主要正文推荐 16fp，行高保持 1.35 到 1.5。
- 统计数字可以大，但不能压过主要任务入口。

## 2. Skymap 的视觉方向

Skymap 应采用“温和、克制、内容优先”的移动工具风格。

推荐气质：

- 安静：少用大面积深色块。
- 清楚：主次动作明确，不让用户猜。
- 可信：状态、失败、导出位置和同步结果都给反馈。
- 轻量：移动端快速完成复盘，不模拟桌面复杂工具。

避免气质：

- 像后台系统一样密集。
- 像临时 Demo 一样到处是手写按钮。
- 像营销页一样用大面积装饰色和夸张标题。
- 把所有能力都作为一级按钮展示。

## 3. 色彩规则

### 3.1 主色

主色应为低饱和雾蓝 / 蓝灰，不使用接近纯黑的按钮色。雾蓝比灰绿更偏现代工具气质，也更适合摄影复盘中的“冷静判断”和“内容优先”。

当前 Harmony 端主色参考：

- `primary`: `#426A7E`
- `primaryPressed`: `#315266`
- `primarySoft`: `#E8F1F5`
- `primaryDisabled`: `#B9CDD7`

使用规则：

- 主色只用于当前页面唯一主动作、关键统计强调和选中状态。
- 同一屏尽量只有一个实心主按钮。
- 主按钮不要用于普通导航、筛选、取消、测试连接等次级动作。

### 3.2 背景与卡片

推荐：

- 页面背景使用暖灰白。
- 卡片表面使用接近白色。
- 卡片边框使用低对比细线。
- 次级容器使用浅灰米色。

避免：

- 大面积黑色或深灰色按钮。
- 多层卡片嵌套。
- 过强阴影、渐变、装饰光斑。
- 用颜色单独表达状态，必须配合文字。

### 3.3 危险色

危险动作使用红色文字或描边，不默认使用大面积实心红。

使用场景：

- 删除复盘记录。
- 清空不可恢复数据。
- 断开或重置配置。

规则：

- 危险动作必须和主动作保持空间距离。
- 危险动作必须有确认弹窗。
- 删除按钮不应和“编辑、导出、保存”同层级展示。

## 4. 字体与层级

### 4.1 推荐字号

- 页面标题：26fp
- 页面副标题：15fp
- 分区标题：18fp
- 卡片标题：17fp
- 正文：16fp
- 辅助说明：15fp
- 元信息：13fp
- 按钮：16fp
- 小按钮：15fp
- 底部导航文字：12fp

规则：

- 正文低于 15fp 前必须有明确理由。
- 不使用 36fp 以上的字作为常规页面标题。
- 统计数字可大一档，但必须服务于“看见积累”，不能抢走主要行动。
- 同一层级字号必须复用 token，不在页面里写散落数字。

### 4.2 行高

- 正文行高保持 22vp 左右。
- 辅助文字行高保持 18 到 20vp。
- 长文本优先换行，不为了卡片高度强行截断。

## 5. 按钮系统

### 5.1 主按钮

用于页面唯一主行动。

示例：

- 开始复盘
- 保存
- 继续编辑
- 进入配置
- 导出

规则：

- 高度 48vp。
- 圆角 8vp。
- 宽度优先撑满容器。
- 同屏只保留一个最强实心主按钮。
- 异步中必须进入禁用态，并改变文案，例如“保存中...”。

### 5.2 次按钮

用于辅助行动。

示例：

- 查看详情
- 测试连接
- 清空搜索与筛选
- 编辑

规则：

- 高度不低于 44vp。
- 使用浅底、描边或轻量背景。
- 不使用主色实心填充。
- 可与主按钮并列，但视觉权重必须低于主按钮。

### 5.3 文本按钮

仅用于极低频或弹层中的轻操作。

规则：

- 不用于关键流程提交。
- 不用于危险动作。
- 不和输入框、卡片标题混在一起造成误触。

### 5.4 危险按钮

用于不可逆操作。

规则：

- 默认红色文字或描边。
- 不使用实心红作为常态。
- 必须二次确认。
- 必须说明后果，例如“删除后无法恢复”。

## 6. 页面结构

### 6.1 首页

目标：让用户继续下一次复盘，并看见积累。

顺序：

1. 页面标题和一句说明。
2. 开始复盘主入口。
3. 复盘概览。
4. 最近一次。
5. 当前状态。

规则：

- 不把统计数据放到主入口之前。
- 不把同步、设置、导出等低频动作放到首页一级按钮。
- 首页统计为辅助反馈，不是仪表盘。

### 6.2 复盘库

目标：让用户找回、筛选、打开历史复盘。

规则：

- 搜索框和筛选 chip 放在列表前。
- 筛选 chip 高度不低于 44vp。
- 删除动作通过滑动或更多菜单触发，并二次确认。
- 空状态必须提供创建第一条复盘入口。

### 6.3 编辑页

目标：让用户顺畅完成复盘文本输入。

规则：

- 输入项必须有可见标签。
- 键盘弹出时，当前输入框必须滚动到可见区域。
- 底部保存按钮不能被键盘或系统手势区遮挡。
- 长文本输入必须保留足够高度。

### 6.4 预览页

目标：阅读复盘，并选择编辑或导出。

规则：

- 顶层只保留“编辑”和“导出”。
- 导出 review.json、上传家庭存储、导出图片、复制复盘数据进入底部弹层。
- 阅读内容优先，按钮区不超过一行。
- 导出图永远不包含页面按钮、状态栏、Toast、页面标题。

### 6.5 设置与同步页

目标：让用户理解当前配置，并完成保存或测试。

规则：

- 状态先行，表单其次，操作最后。
- 测试连接是次按钮，保存是主按钮。
- 失败必须说明原因或下一步。
- 密码、凭据等字段不和普通字段混淆文案。

## 7. 弹层与菜单

底部弹层用于承载同一主题下的多个低频动作。

适用：

- 导出格式选择。
- 分享或复制方式。
- 更多操作。

规则：

- 弹层标题必须明确。
- 动作按使用频率排序。
- 取消操作放在底部，并与主动作组分开。
- 背景遮罩必须足够明显。
- 点击遮罩可关闭，但危险流程不能只靠遮罩退出。

## 8. 反馈与状态

必须反馈：

- 保存成功或失败。
- 导出成功、导出位置或失败兜底。
- 上传成功或失败。
- 测试连接结果。
- 历史记录读取失败。

规则：

- Toast 用于短反馈。
- 会影响用户下一步判断的结果，需要在页面内留下可见状态。
- 加载超过 300ms 的动作必须改变按钮文案或禁用态。
- 不允许按钮看起来可点但点击后没有任何反馈。

## 9. 空状态

空状态应包含：

1. 当前为什么为空。
2. 用户下一步能做什么。
3. 一个合适行动。

示例：

- 复盘库为空：提示“可先创建第一条复盘”，并提供“创建第一条复盘”。
- 搜索无结果：提示换关键词或切换筛选，并提供“清空搜索与筛选”。
- 首页无最近记录：提示开始第一条复盘后会展示最近内容。

## 10. 实现规则

### 10.1 Token 优先

所有新增 UI 必须优先使用：

- `AppColors`
- `AppMetrics`
- `AppTypography`
- `PrimaryButton`
- `SecondaryButton`

不允许在页面中随意新增 raw hex、字号和按钮高度。确需新增时，先扩展公共 token。

### 10.2 页面内禁止重复造按钮

除非有明确交互差异，否则不得在页面里手写一套新的主按钮、次按钮或危险按钮。

允许例外：

- 预览页顶部小型动作按钮。
- 筛选 chip。
- 系统控件无法被公共组件覆盖的场景。

例外必须保持高度、字号、颜色和按压态与本规范一致。

### 10.3 验证

涉及 UI 的改动至少验证：

- 是否仍无显式无意义返回按钮。
- 主按钮是否同屏唯一。
- 触控高度是否不低于 44vp。
- 空状态是否有下一步。
- 异步按钮是否有禁用态。
- 构建是否通过。

Harmony 端当前可运行：

```bash
node scripts/verify_home_stats.mjs
node scripts/verify_compact_ui_tokens.mjs
node scripts/verify_home_information_architecture.mjs
node scripts/verify_review_library_v11.mjs
DEVECO_SDK_HOME=/Applications/DevEco-Studio.app/Contents/sdk JAVA_HOME=/Applications/DevEco-Studio.app/Contents/jbr/Contents/Home /Applications/DevEco-Studio.app/Contents/tools/hvigor/bin/hvigorw --mode module -p module=entry -p product=default assembleHap --no-daemon
```

## 11. 评审清单

每次 UI PR 评审时问：

- 这个页面的唯一核心目标是什么？
- 用户第一眼能不能知道下一步？
- 主按钮是不是只有一个？
- 次按钮有没有抢主按钮？
- 危险动作有没有降权和确认？
- 页面是否依赖系统返回，而不是自己重复放返回按钮？
- 字号、行高、颜色是否来自公共 token？
- 空状态、失败状态、加载状态是否完整？
- 导出和同步是否说明结果位置和失败恢复路径？

如果任一项回答不清楚，先改设计，再继续实现。
