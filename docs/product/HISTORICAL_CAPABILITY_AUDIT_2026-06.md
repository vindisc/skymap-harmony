# Historical Capability Recovery Audit 2026-06

日期：2026-06-13

范围：`skymap-HarmonyOS`

结论：**B 暂缓**

本审计只回答一件事：Harmony 历史分支里的 SMB / WebDAV / 家庭存储能力，为什么出现、为什么消失、现在是否值得恢复。

---

## 0. 审计结论摘要

- 这些能力**确实在历史分支里实现过**，不是空文档，也不是只做了产品设想。
- 当前 `HEAD` **没有这些代码和入口**，但它们的消失更接近**实验分支未合线**，不是主线后续又专门做了一轮主动删除。
- 这批能力的产品方向**仍然符合 Skymap 的长期方向**，但**不适合在当前 Beta 阶段恢复**。
- 如果以后恢复，建议优先恢复 **WebDAV / 家庭存储配置能力**，而不是直接恢复 **SMB 直连上传**。

---

## 1. 历史能力清单

### 1.1 WebDAV

历史提交：

- `de1a179` 新增家庭存储 WebDAV 上传

历史能力：

- 设置页可填写 WebDAV 地址、用户名、密码或 Token、远程目录。
- 应用可测试 WebDAV 连接。
- 阅读页可把复盘 JSON 直接上传到 WebDAV 目录。

历史入口：

- 设置页：`entry/src/main/ets/pages/ReviewSettingsPage.ets`
- 服务：`entry/src/main/ets/services/HomeStorageService.ets`
- 阅读页入口：`entry/src/main/ets/pages/PreviewPage.ets`

备注：

- `HomeStorageService` 使用 `@ohos.net.http` 发起 WebDAV 请求。
- 文件目标是 `*.review.json` 或同类复盘交换 JSON。

### 1.2 SMB 系统保存模式

历史提交：

- `8e711b1` 新增 SMB 系统保存模式

历史能力：

- 设置页可切换到 SMB 模式。
- 用户填写 SMB 地址、端口、共享名、远程目录、用户名和密码。
- 该阶段的 SMB 更像“借助系统保存面板完成保存”的过渡方案，不是完整的低层协议直连。

历史入口：

- 设置页：`entry/src/main/ets/pages/ReviewSettingsPage.ets`
- 服务：`entry/src/main/ets/services/HomeStorageService.ets`

备注：

- 这是从 WebDAV 向“家庭存储统一入口”扩展的第一步。
- 说明当时产品目标已经从“只导出 JSON”推进到“直接把 JSON 送进家庭存储”。

### 1.3 SMB 直连上传

历史提交：

- `a90cea7` 实现 SMB 直连上传

历史能力：

- 应用内直接连接 SMB 共享目录。
- 支持主机、端口、共享名、远程目录、用户名、密码、工作组或域。
- 阅读页可以把复盘 JSON 直接写入 SMB 共享。

历史入口：

- 设置页：`entry/src/main/ets/pages/ReviewSettingsPage.ets`
- 服务：`entry/src/main/ets/services/HomeStorageService.ets`
- 协议实现：`entry/src/main/ets/services/Smb2Client.ets`
- 阅读页入口：`entry/src/main/ets/pages/PreviewPage.ets`

备注：

- `Smb2Client.ets` 是超过千行的低层协议实现，不是简单封装。
- 这说明当时不是停留在“支持 SMB”口头层面，而是实际下探到了 SMB2 / NTLM 协议细节。

### 1.4 家庭存储安全与设置增强

历史提交：

- `33ba91a` 优化阅读页家庭存储入口
- `c02e44a` 加固家庭存储安全与设置体验

历史能力：

- 阅读页把“导出复盘文件”和“上传家庭存储”并列展示。
- 设置页支持更完整的家庭存储表单。
- 密码不再只放本地 preferences，而是引入安全存储。
- 设置页开始处理键盘遮挡、滚动和输入聚焦体验。

历史入口：

- 阅读页：`entry/src/main/ets/pages/PreviewPage.ets`
- 设置页：`entry/src/main/ets/pages/ReviewSettingsPage.ets`
- 安全存储：`entry/src/main/ets/services/HomeStorageSecretService.ets`

备注：

- `c02e44a` 还新增了 `scripts/verify_security_hardening.mjs`。
- 这说明当时已经意识到“家庭存储不是只要能传文件就算完成”，还涉及密码安全和移动端设置体验。

### 1.5 当前状态

当前 `HEAD` 中：

- 没有 `HomeStorageService.ets`
- 没有 `Smb2Client.ets`
- 没有 `HomeStorageSecretService.ets`
- 设置页只保留“复盘人”
- 阅读页只保留 `导出 review.json / 复制复盘数据 / 导出`

结论：

- 历史能力**曾经存在**
- 当前基线**已经不存在**

## 2. 为什么会出现

从提交链和历史文档看，这批能力出现有三个直接原因。

### 2.1 产品方向需要一个“家庭存储中转”

历史 `docs/mobile-main-flow.md` 在 `c02e44a` 时明确写过：

- 复盘设置包含家庭存储配置
- 设置页可选 `WebDAV` 或 `SMB`
- 阅读页“上传家庭存储”会直接上传 `*.review.json`

这说明当时的目标不是单纯导出文件，而是尝试建立：

```text
Harmony
↓
家庭存储
↓
Mac
```

的接力链路。

### 2.2 当时在探索比“手动找文件”更顺的跨端交换

仅靠应用沙箱导出路径，用户很难真正拿到文件；而家庭存储思路天然满足：

- 不依赖云账号
- 符合 NAS / 私有网盘场景
- 让手机和 Mac 通过共享目录接力

所以这些实现本质上是在提前试探后来的 `Sync System v1` 产品方向。

### 2.3 当时的复盘对象仍然是单文件 JSON

这批能力上传的核心对象是复盘 JSON，而不是完整 bundle、manifest 或照片副本。

这使得它们在当时具备几个吸引力：

- 文件小
- 网络负担低
- 容易落地第一版
- 不需要先解决照片二进制同步

所以它们出现得很快，也比较适合在实验分支里快速验证。

---

## 3. 为什么会消失

### 3.1 主要原因：合线丢失，而不是主线主动删除

证据很清楚：

- 历史链路是  
  `6ad033d -> f7d4c59 -> de1a179 -> 8e711b1 -> a90cea7 -> 33ba91a -> c02e44a`
- 当前 Beta 主线则是从 `6ad033d` 走向  
  `ae4cc75 -> d958fe4 -> ce3f834`

这两条线的共同祖先都是 `6ad033d`，但历史家庭存储链路没有进入当前主线。

所以当前代码里看不到 SMB / WebDAV，不是因为后来又有一个“删除家庭存储能力”的明确提交，而是因为：

- 家庭存储能力停留在历史实验线
- 当前 Beta 主线选择了另一条更短的 `review.json` 手动交换路线

### 3.2 次要原因：当前 Beta 明确收口到最小可用链路

当前 Beta 目标强调的是：

```text
Harmony 导出 review.json
↓
用户手动交给 Mac
↓
Mac 导入 Review Library
```

这条链路的优先级高于：

- 家庭存储自动接力
- SMB / WebDAV 上传
- 更复杂的同步设置页

也就是说，产品重心从“继续扩同步”转成了“先确保用户真的能完成一次交换”。

### 3.3 复杂度上升，实验线开始暴露额外成本

`c02e44a` 已经暴露出三类额外成本：

- 密码不能继续裸存，需要安全存储
- 设置页字段多，必须处理键盘遮挡和输入体验
- 上传动作要处理忙碌状态、错误反馈和连接测试

这说明家庭存储能力并没有停留在“再补两个按钮”的阶段，而是已经开始拖出一整套：

- 安全问题
- 表单问题
- 网络问题
- 真机兼容问题

对 Beta 收口来说，这类成本很容易压过它带来的收益。

### 3.4 没有证据表明它完成了主线级真机验收

历史提交里能看到本地验证脚本，但看不到当前主线已经接受这批能力的证据。

因此更合理的判断是：

- 这是一次真实实现过的实验
- 但实验没有走到“纳入当前 Beta 主线”的阶段

## 4. 恢复成本

### 4.1 WebDAV：中

原因：

- 历史上已有完整服务和设置入口。
- 协议层比 SMB 直连简单。
- 仍需适配当前 `review.json` 导出链路、当前页面结构、当前设置页收口方式。
- 仍需补真机验证、失败提示、权限与目录错误处理。

判断：

- 不是从零开始
- 但也不是直接 cherry-pick 就能稳定上线

### 4.2 SMB 系统保存模式：中

原因：

- 比 SMB 直连更轻。
- 更接近“用户看得见文件结果”的当前产品要求。
- 但当前主线已经转向系统文件导出，SMB 系统保存模式是否还保留独立价值，需要重新定义。

判断：

- 可作为过渡方案考虑
- 但产品收益未必足以支撑恢复成本

### 4.3 SMB 直连上传：高

原因：

- `Smb2Client.ets` 体量大，协议细节重。
- 涉及 SMB2、认证、共享目录兼容性、NAS 差异和网络错误处理。
- 真机和不同 NAS 的回归成本会远高于 WebDAV。
- 一旦出问题，用户几乎无法自助诊断。

判断：

- 技术债和维护成本都高
- 不适合作为 Beta 后第一个恢复目标

### 4.4 家庭存储整链恢复：高

原因：

- 不是只恢复一个服务文件，而是要一起恢复：
  - 设置页入口
  - 阅读页入口
  - 凭据存储
  - 上传服务
  - 反馈文案
  - 真机验证
- 还要重新确认它与当前 `Sync v0.5`、未来 `Sync v1` 的边界关系。

判断：

- 如果按“产品级能力”恢复，整体成本应按高评估

## 5. 当前价值

### 5.1 是否仍符合 Skymap Product 方向

结论：**符合长期方向，不符合当前 Beta 优先级**

原因：

- Skymap 仍然需要一个不依赖账号和云后端的跨端接力方案。
- 家庭存储、NAS、共享目录、本地文件夹，仍然是合理方向。
- `Sync System v1` 的产品设计仍然把 SMB / WebDAV 视为家庭存储入口。

但同时也要承认：

- 当前 Beta 已经有一条更小、更可验证的 `review.json` 手动交换闭环。
- 对 Beta 用户测试来说，家庭存储不是当前阻塞点。

### 5.2 它今天还能解决什么问题

如果恢复，能解决的问题主要是：

- 少一次“手动发送给 Mac”
- 让有 NAS / WebDAV 环境的用户更顺地接力
- 为未来 `Sync v1` 铺一条更接近真实同步的入口

### 5.3 它今天不能解决什么问题

即使恢复，也仍然不能直接解决：

- 当前 Beta 最核心的可理解性问题
- 照片与复盘的稳定配对
- review bundle / manifest 边界
- Mac 端自动扫描和稳定导入

所以它的价值更偏向“后续增强”，不是“当前 Beta 必需品”。

---

## 6. 恢复建议

结论：**B 暂缓**

理由：

### 6.1 不值得在当前 Beta 恢复

当前 Beta 的成功标准聚焦于：

- Harmony 创建复盘
- 导出 `review.json`
- Mac 导入 Review Library
- 继续编辑与导出

家庭存储链路不在这条最小闭环里。

### 6.2 值得保留为 Beta 后方向

虽然当前不该恢复，但也不建议直接判死刑，因为：

- 它仍符合 Skymap 的长期产品方向
- 历史上已经证明团队愿意为这条路径投入实现
- WebDAV 和家庭存储配置仍可能成为 `Sync v1` 的可用起点

### 6.3 如果以后恢复，建议顺序如下

1. 优先审查历史 WebDAV 能力是否可重用。
2. 重新定义家庭存储能力与 `Sync v0.5` / `Sync v1` 的边界。
3. 只在明确目标后再决定是否恢复 SMB。
4. SMB 若恢复，优先考虑更可维护的方案，不建议直接恢复低层 SMB 直连实现。

---

## 7. Beta 影响

### 7.1 如果不恢复，是否影响当前 Beta

结论：**不影响当前 Beta**

原因：

- 当前 Beta 主链路已经定义为 `review.json` 手动交换。
- SMB / WebDAV 不在当前 Beta 成功标准中。
- 当前 Beta 的阻塞问题是“用户拿不拿得到文件”和“Mac 能不能独立导入”，不是“能不能自动上传到家庭存储”。

### 7.2 如果错误恢复，反而会带来什么风险

- 稀释当前 Beta 范围
- 把验证焦点从主链路拉走
- 引入网络、权限、凭据和 NAS 兼容性问题
- 让用户测试任务重新变复杂

因此，从 Beta 策略看，当前最佳动作不是恢复，而是：

- 先把历史事实记录清楚
- 后续在 Beta 后按独立专题重评

---

## 8. 最终判断

本轮 Historical Capability Recovery Audit 的最终判断如下：

- **为什么出现**：因为当时在探索“Harmony -> 家庭存储 -> Mac”的低后端依赖接力方案，且以单文件复盘 JSON 为最小落地对象。
- **为什么消失**：主要不是主动删除，而是家庭存储实验线没有合入当前 Beta 主线；主线改走了更小的 `review.json` 手动交换路线。
- **是否值得恢复**：长期看值得保留方向，短期看不值得进入当前 Beta。
- **建议结论**：**B 暂缓**

附注：

- WebDAV：可作为后续恢复的优先候选。
- SMB 系统保存模式：可保留为备选过渡方案。
- SMB 直连上传：维护成本高，不建议优先恢复。
